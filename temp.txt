# 🧠 Smriti-OS: Deep Architecture Analysis & Development Roadmap
**Date:** April 7, 2026 | **Analyst:** Antigravity AI | **Status:** CRITICAL REVIEW

---

## 📊 Executive Summary

Smriti-OS has a **strong architectural foundation** but is at **~30% functional parity** with Shoper 9. The core "Memory, Not Code" design pattern is correctly established. The critical gap is in the **Application Layer** — the database schema is solid, but the UI components, service layer, and business logic are largely incomplete or stubbed.

> [!IMPORTANT]
> The `DynamicGrid.tsx` component has a critical `handleSave` that contains `// TODO: Write actual save logic`. This means **no data is actually persisted** in PO or GRN modules currently. This is the most urgent defect to fix.

### 4. Schema Duality Problem 🚨

**Critical Architectural Conflict Detected:**

There are **TWO different database schemas**:
1. `001_initial_retail_procurement.sql` → PostgreSQL syntax (`UUID`, `gen_random_uuid()`, `JSONB`, triggers in `plpgsql`)
2. `db.ts` → SQLite/WASM via `sql.js` (uses `TEXT` PKs, `INTEGER DEFAULT`, no `UUID`)

The app actually runs on **SQLite (sql.js)**, but the SQL file is written in **PostgreSQL**. The schema in `db.ts::syncSchema()` is what actually runs. The `001_initial_retail_procurement.sql` is served as a static file (`/001_retail_schema.sql`) but it contains PostgreSQL syntax that **will fail silently** in SQLite.

> [!WARNING]
> **This is a silent data corruption risk.** The `populateInitialSchema()` fetches `/001_retail_schema.sql` and runs it. But `gen_random_uuid()`, `JSONB`, and `plpgsql` triggers **do not exist in SQLite**. The initial schema likely silently fails and only `syncSchema()`'s SQLite-compatible tables are actually created.

**Recommendation:** Either:
- **(A) Create a SQLite-native schema file** (`/001_retail_schema_sqlite.sql`) that matches what `syncSchema()` creates, or
- **(B) Remove `populateInitialSchema()`** and let `syncSchema()` be the single source of truth.

The entire Procurement module currently depends on a table that may not exist.

## 🎯 The Single Most Impactful Suggestion

**Build the `ConfigHub` screen first.**

Here's why: Every feature from Phase 3 onwards (Billing, Stock Validation, Size-wise) requires specific `system_parameters` values to work correctly. Without a Config Hub, developers must manually run SQL inserts every time a business rule needs to change. With a Config Hub:
- The store owner can self-configure their own POS behavior
- Development moves faster (test different modes without code changes)
- It's the defining feature that separates Smriti-OS from generic ERPs

The DNA of the entire "Memory, Not Code" philosophy is realized in this single screen.
